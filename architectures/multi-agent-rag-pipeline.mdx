## Problem & Context

Legal teams processing thousands of contracts need fast, accurate extraction of key clauses, obligations, and risk factors. Manual review is slow and error-prone. A single LLM call cannot reliably handle documents exceeding 100 pages with cross-referenced annexes.

## Architecture Overview

The pipeline uses three specialized agents coordinated by an orchestrator. Documents are chunked, embedded, and stored in a vector database. Each agent queries only the chunks relevant to its domain: clause extraction, obligation mapping, or risk assessment. A human review gate sits between agent output and final delivery.

## Components

- **Orchestrator Agent**: Routes incoming documents, manages agent lifecycle, and aggregates results.
- **Clause Extraction Agent**: Identifies and classifies contract clauses using few-shot prompting against a curated taxonomy.
- **Obligation Mapper Agent**: Traces obligations across document sections and links them to responsible parties.
- **Risk Assessment Agent**: Flags non-standard terms and deviations from template language.
- **Vector Store**: pgvector on PostgreSQL, with per-document namespace isolation.
- **Review Dashboard**: Web UI where legal reviewers approve, reject, or modify agent findings before export.

## Data Flow

Documents are ingested as PDF, converted to markdown, chunked at 512 tokens with 64-token overlap, and embedded using a local model. The orchestrator dispatches tasks in parallel. Each agent retrieves relevant chunks, processes them, and writes structured JSON to a shared results store. The review dashboard polls for completed batches.

## Trade-offs & Decisions

We chose pgvector over a dedicated vector database to reduce infrastructure complexity. Chunk size was tuned empirically; 512 tokens balanced retrieval precision against context coverage. Human review gates add latency but are non-negotiable for legal compliance.

## Lessons Learned

Parallel agent execution cut processing time by 60% compared to sequential runs. Namespace isolation in the vector store prevented cross-document contamination that caused early hallucinations. Investing in a strong evaluation dataset upfront saved weeks of debugging downstream.
