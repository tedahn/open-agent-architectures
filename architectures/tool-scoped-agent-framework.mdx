## Problem & Context

DevOps teams managing 50+ services face a dilemma: giving an agent access to all tools creates safety risks and confuses the model, while static tool sets cannot adapt to varied tasks like incident response, deployments, and infrastructure provisioning.

## Architecture Overview

The framework uses a two-phase approach. A lightweight planner agent receives the task description and selects a scoped tool set from a registry. The executor agent then operates with only those tools available. A policy engine enforces hard constraints such as blocking production database writes during business hours.

## Components

- **Planner Agent**: Analyzes task intent and selects tools from the registry. Uses a fine-tuned classifier to map task descriptions to tool categories.
- **Tool Registry**: A versioned catalog of all available tools with metadata including risk level, required approvals, and blast radius.
- **Policy Engine**: Evaluates tool selections against organizational rules before granting access. Supports time-based, role-based, and environment-based constraints.
- **Executor Agent**: Performs the actual work with the scoped tool set. Runs in a sandboxed environment with resource limits.
- **Audit Logger**: Records every tool invocation, including inputs, outputs, and the policy evaluation trace.

## Data Flow

A task enters the planner, which queries the tool registry and proposes a tool set. The policy engine validates the proposal, potentially removing high-risk tools or requiring human approval. The approved tool set is injected into the executor agent's context. All actions are logged to an append-only audit store.

## Trade-offs & Decisions

Dynamic scoping adds one LLM call of latency compared to static tool sets, but reduces average tool count from 50+ to 5-8 per task, improving both safety and response quality. We chose an allowlist approach over a blocklist because new tools default to restricted access.

## Lessons Learned

Tool descriptions matter more than tool count. Well-written descriptions reduced mis-selection by 40%. The policy engine caught 12 potentially dangerous operations in the first month of production use. Versioning the tool registry proved essential when rolling back breaking changes to tool interfaces.
